## Architecture Recommendation

For a fast-growing food delivery startup handling diverse data types (GPS logs, text reviews, payment transactions, menu images), I would recommend a **Data Lakehouse** architecture, and here are three compelling reasons why this hybrid approach outperforms traditional data warehouses or pure data lakes.

**Reason 1: Multi-Format Data Requires Flexible Storage**. A traditional data warehouse expects structured, relational data with fixed schemas—it's built for payment transactions and order tables, not raw GPS coordinates streaming every 30 seconds or unstructured customer reviews. A pure data lake can store everything but lacks the query performance and ACID guarantees needed for real-time operational analytics. The lakehouse combines both: it stores raw GPS logs, images, and reviews in cheap object storage (like S3) using open formats like Parquet, while simultaneously supporting fast SQL queries via metadata layers and indexing. This means data scientists can access raw GPS streams for route optimization models while business analysts run SQL dashboards on the same storage—without duplicating data into a separate warehouse.

**Reason 2: Real-Time Analytics with Schema Evolution**. Food delivery operations demand real-time insights: which restaurants are delayed, which drivers are near customers, what menu items are trending right now. A data warehouse requires upfront schema definition and ETL pipelines that add latency—by the time GPS data is cleaned, loaded, and aggregated, the delivery window has passed. A lakehouse enables schema-on-read: dump raw GPS JSON directly into storage, then query it immediately with DuckDB or Spark. As the business evolves (adding drone deliveries, subscription tiers, dynamic pricing), the lakehouse schema can adapt without rewriting pipelines. The startup doesn't get locked into a rigid warehouse model that becomes outdated as product features change.

**Reason 3: Cost Efficiency at Scale**. Storing high-volume, low-value data like detailed GPS traces (millions of coordinates per day) in a traditional warehouse is prohibitively expensive due to compute-coupled storage costs. A data lake offers cheap storage but lacks governance, versioning, and transaction support—leading to "data swamps" where analysts can't trust data quality. The lakehouse provides warehouse-like reliability (ACID transactions via Delta Lake or Iceberg) at lake-scale economics. Raw menu images and full GPS traces live in $0.02/GB S3 storage, while critical payment transactions get indexed and cached for sub-second queries. The startup pays for storage and compute independently, scaling each based on actual needs rather than warehouse licensing tiers.

In summary, a data lakehouse gives this food delivery startup the storage flexibility of a lake, the query performance of a warehouse, and the cost structure needed to scale from thousands to millions of daily deliveries without architectural rewrites.
